# Alert Pipeline Documentation

This document describes the complete alerting pipeline for the URL Shortener production system, from metric collection through alert delivery. Every component described here has been verified end-to-end with real alerts captured during chaos engineering experiments on 2026-04-05.

---

## 1. Pipeline Architecture

```
                        15s scrape interval
Prometheus (port 9090) ─────────────────────> Flask app1:5000/metrics
         │                                    Flask app2:5000/metrics
         │                                    Flask app3:5000/metrics
         │             15s scrape interval
         │ ───────────────────────────────> node-exporter:9100/metrics
         │
         │  evaluates alert_rules.yml every 15s (evaluation_interval)
         │
         ├─ Rule matches? ─> pending ─(for duration)─> firing
         │
         v
Alertmanager (port 9093)
         │
         │  Groups alerts by [alertname, severity]
         │  Routes:
         │    severity=critical -> group_wait=5s,  repeat_interval=2m
         │    severity=warning  -> group_wait=15s, repeat_interval=5m
         │
         │  Inhibition: critical suppresses matching warning alerts
         │
         v
Webhook Receiver (port 9094)
         │
         │  POST /alert
         │  Parses Alertmanager JSON payload
         │
         ├─> /var/log/alerts.log          (append-only structured log, one JSON line per batch)
         └─> /app/evidence/alert_*.json   (one file per individual alert, named by alert+status+timestamp)
```

## 2. Component 1: Prometheus Metric Collection

### Configuration

**File:** `prometheus/prometheus.yml`

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

rule_files:
  - "alert_rules.yml"

alerting:
  alertmanagers:
    - static_configs:
        - targets:
            - alertmanager:9093

scrape_configs:
  - job_name: "flask-app"
    scrape_interval: 15s
    static_configs:
      - targets:
          - "app1:5000"
          - "app2:5000"
          - "app3:5000"
    metrics_path: /metrics

  - job_name: "node-exporter"
    scrape_interval: 15s
    static_configs:
      - targets:
          - "node-exporter:9100"
    metrics_path: /metrics
```

### Key Parameters

| Parameter | Value | Effect |
|---|---|---|
| `global.scrape_interval` | 15s | Default interval for all jobs unless overridden |
| `global.evaluation_interval` | 15s | How often Prometheus evaluates alert rules |
| `flask-app.scrape_interval` | 15s | Matches global interval; Flask apps scraped every 15 seconds |
| `node-exporter.scrape_interval` | 15s | System metrics collected at default interval |

### Scraped Metrics

The Flask application exposes metrics at `/metrics` via `prometheus_client` (Python). Key metrics:

| Metric | Type | Labels | Description |
|---|---|---|---|
| `up` | Gauge | `job`, `instance` | Automatically generated by Prometheus; 1 if scrape succeeds, 0 if it fails |
| `http_requests_total` | Counter | `method`, `endpoint`, `status` | Total HTTP requests handled |
| `http_request_duration_seconds` | Histogram | `method`, `endpoint` | Request latency with buckets: 5ms, 10ms, 25ms, 50ms, 100ms, 250ms, 500ms, 1s, 2.5s, 5s, 10s |
| `urls_created_total` | Counter | (none) | Business metric: total URLs shortened |
| `redirects_total` | Counter | (none) | Business metric: total redirects served |
| `cache_hits_total` | Counter | (none) | Redis cache hits |
| `cache_misses_total` | Counter | (none) | Redis cache misses |
| `active_urls` | Gauge | (none) | Current count of active shortened URLs |
| `process_resident_memory_bytes` | Gauge | (none) | RSS memory of gunicorn process (from prometheus_client default) |

### Docker Compose Deployment

```yaml
prometheus:
  image: prom/prometheus:latest
  ports:
    - "9090:9090"
  volumes:
    - ./prometheus/prometheus.yml:/etc/prometheus/prometheus.yml:ro
    - ./prometheus/alert_rules.yml:/etc/prometheus/alert_rules.yml:ro
    - prometheus_data:/prometheus
  command:
    - "--config.file=/etc/prometheus/prometheus.yml"
    - "--storage.tsdb.retention.time=7d"
  restart: unless-stopped
```

## 3. Component 2: Alert Rules

### Configuration

**File:** `prometheus/alert_rules.yml`

Two alert groups are defined: application-level and infrastructure-level.

### Application Alert Rules (Group: `url-shortener-alerts`)

#### ServiceDown

```yaml
- alert: ServiceDown
  expr: up{job="flask-app"} == 0
  for: 15s
  labels:
    severity: critical
  annotations:
    summary: "Flask instance {{ $labels.instance }} is down"
    description: "{{ $labels.instance }} has been unreachable for more than 15 seconds."
```

**Behavior:** Fires when Prometheus fails to scrape a Flask instance for 15 consecutive seconds. The `up` metric is automatically set to 0 when a scrape target is unreachable. With a 10-second scrape interval, the alert transitions to firing after at minimum 2 failed scrapes (20 seconds) plus the `for` duration holdoff.

**Verified:** Alert fired for `app2:5000` at 02:44:06Z and `app5:5000` at 02:49:21Z during chaos experiments (app5 was part of a temporary 5-instance configuration used during chaos testing; the production architecture runs 3 instances). Both firing and resolved alerts were captured.

#### HighErrorRate

```yaml
- alert: HighErrorRate
  expr: |
    (
      sum(rate(http_requests_total{status=~"5.."}[5m])) by (instance)
      /
      sum(rate(http_requests_total[5m])) by (instance)
    ) > 0.10
  for: 2m
  labels:
    severity: warning
  annotations:
    summary: "High error rate on {{ $labels.instance }}"
    description: "Error rate is {{ $value | humanizePercentage }} over the last 5 minutes."
```

**Behavior:** Fires when any single instance's 5xx error rate exceeds 10% of total requests, sustained for 2 minutes. Uses a 5-minute rate window to smooth out brief spikes.

#### HighLatency

```yaml
- alert: HighLatency
  expr: |
    histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, instance)) > 2.0
  for: 3m
  labels:
    severity: warning
  annotations:
    summary: "High latency on {{ $labels.instance }}"
    description: "p95 latency is {{ $value }}s, exceeding 2s threshold."
```

**Behavior:** Fires when the p95 request latency on any instance exceeds 2 seconds for 3 minutes. The 3-minute `for` duration prevents alerting on brief latency spikes during garbage collection or connection pool churn.

#### HighMemoryUsage

```yaml
- alert: HighMemoryUsage
  expr: process_resident_memory_bytes / 1024 / 1024 > 512
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "High memory usage on {{ $labels.instance }}"
    description: "Memory usage is {{ $value }}MB."
```

**Behavior:** Fires when a Flask instance's RSS memory exceeds 512MB for 5 minutes. Intended as an early warning before OOM conditions.

### Infrastructure Alert Rules (Group: `infrastructure-alerts`)

#### NodeExporterDown

```yaml
- alert: NodeExporterDown
  expr: up{job="node-exporter"} == 0
  for: 30s
  labels:
    severity: critical
```

**Behavior:** Fires when the Node Exporter cannot be scraped for 30 seconds. Indicates the host-level monitoring agent is down.

#### HostHighCpuUsage

```yaml
- alert: HostHighCpuUsage
  expr: 100 - (avg by(instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
  for: 5m
  labels:
    severity: warning
```

**Behavior:** Fires when average CPU utilization exceeds 80% for 5 minutes. Uses idle-time inversion for accuracy.

**Verified:** Alert fired at 02:50:49Z when CPU usage reached 96.68% during chaos experiments (multiple container restarts under load). Evidence captured in `alert_HostHighCpuUsage_firing_2026-04-05T02-51-04.895969Z.json`.

#### HostHighMemoryUsage

```yaml
- alert: HostHighMemoryUsage
  expr: (1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85
  for: 5m
  labels:
    severity: warning
```

#### HostDiskSpaceLow / HostDiskSpaceCritical

```yaml
- alert: HostDiskSpaceLow
  expr: (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}) * 100 < 15
  for: 5m
  labels:
    severity: warning

- alert: HostDiskSpaceCritical
  expr: (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}) * 100 < 5
  for: 1m
  labels:
    severity: critical
```

**Behavior:** Two-tier disk space alerting. Warning at 15% free (5-minute hold), critical at 5% free (1-minute hold). The inhibition rule ensures that when the critical alert fires, the warning alert is suppressed.

#### HostNetworkErrors

```yaml
- alert: HostNetworkErrors
  expr: rate(node_network_receive_errs_total[5m]) + rate(node_network_transmit_errs_total[5m]) > 10
  for: 5m
  labels:
    severity: warning
```

### Alert Rule Summary

| Alert | Expression | For | Severity | Group |
|---|---|---|---|---|
| ServiceDown | `up{job="flask-app"} == 0` | 15s | critical | url-shortener-alerts |
| HighErrorRate | 5xx rate > 10% by instance | 2m | warning | url-shortener-alerts |
| HighLatency | p95 > 2s by instance | 3m | warning | url-shortener-alerts |
| HighMemoryUsage | RSS > 512MB | 5m | warning | url-shortener-alerts |
| NodeExporterDown | `up{job="node-exporter"} == 0` | 30s | critical | infrastructure-alerts |
| HostHighCpuUsage | CPU > 80% | 5m | warning | infrastructure-alerts |
| HostHighMemoryUsage | Memory > 85% | 5m | warning | infrastructure-alerts |
| HostDiskSpaceLow | Disk free < 15% | 5m | warning | infrastructure-alerts |
| HostDiskSpaceCritical | Disk free < 5% | 1m | critical | infrastructure-alerts |
| HostNetworkErrors | Error rate > 10/s | 5m | warning | infrastructure-alerts |

## 4. Component 3: Alertmanager Routing

### Configuration

**File:** `alertmanager/alertmanager.yml`

```yaml
global:
  resolve_timeout: 5m

route:
  group_by: ["alertname", "severity"]
  group_wait: 10s
  group_interval: 1m
  repeat_interval: 5m
  receiver: "webhook"

  routes:
    - match:
        severity: critical
      receiver: "webhook"
      group_wait: 5s
      repeat_interval: 2m

    - match:
        severity: warning
      receiver: "webhook"
      group_wait: 15s
      repeat_interval: 5m

receivers:
  - name: "webhook"
    webhook_configs:
      - url: "http://webhook-receiver:9094/alert"
        send_resolved: true

inhibit_rules:
  - source_match:
      severity: critical
    target_match:
      severity: warning
    equal: ["alertname", "instance"]
```

### Routing Behavior

**Grouping:** Alerts are grouped by `alertname` and `severity`. This means all `ServiceDown` critical alerts are batched together, and all `HighErrorRate` warning alerts are batched together. This prevents alert storms when multiple instances fail simultaneously.

**Critical alerts (severity=critical):**
- `group_wait`: 5 seconds. After the first alert in a group arrives, Alertmanager waits 5 seconds before sending, to batch any additional alerts that fire in the same window.
- `repeat_interval`: 2 minutes. If the alert remains firing, a reminder notification is sent every 2 minutes.

**Warning alerts (severity=warning):**
- `group_wait`: 15 seconds. Longer batching window since warnings are less urgent.
- `repeat_interval`: 5 minutes. Less frequent reminders.

**Default route (fallback):**
- `group_wait`: 10 seconds
- `group_interval`: 1 minute (minimum interval between notifications for the same group when new alerts are added)
- `repeat_interval`: 5 minutes

### Inhibition Rules

```yaml
inhibit_rules:
  - source_match:
      severity: critical
    target_match:
      severity: warning
    equal: ["alertname", "instance"]
```

When a `critical` alert is firing for a given `alertname` + `instance` combination, any `warning` alert with the same `alertname` + `instance` is suppressed. This prevents duplicate notifications -- for example, if `HostDiskSpaceCritical` fires for `node-exporter:9100`, the `HostDiskSpaceLow` alert for the same instance is inhibited.

### Resolved Alerts

The `send_resolved: true` configuration on the webhook receiver means Alertmanager sends a notification when an alert transitions from `firing` to `resolved`. This is critical for incident tracking -- it provides an explicit end-of-incident marker with the same `fingerprint` as the firing alert.

The `resolve_timeout: 5m` global setting means that if Prometheus stops sending an alert (e.g., Prometheus itself crashes), Alertmanager will auto-resolve the alert after 5 minutes of silence.

### Docker Compose Deployment

```yaml
alertmanager:
  image: prom/alertmanager:latest
  ports:
    - "9093:9093"
  volumes:
    - ./alertmanager/alertmanager.yml:/etc/alertmanager/alertmanager.yml:ro
  command:
    - "--config.file=/etc/alertmanager/alertmanager.yml"
  restart: unless-stopped
```

## 5. Component 4: Webhook Receiver

### Implementation

**File:** `scripts/webhook-receiver.py`

A custom Python HTTP server (stdlib only, no external dependencies) that receives alert payloads from Alertmanager and persists them in two forms:

1. **Append-only log file** (`/var/log/alerts.log`): One JSON line per alert batch. Contains the raw Alertmanager payload with a receive timestamp. Used for audit trail and historical analysis.

2. **Individual evidence files** (`/app/evidence/alert_<name>_<status>_<timestamp>.json`): One file per individual alert. Contains parsed alert metadata (name, severity, status) plus the full alert object. Used for incident investigation and hackathon evidence.

### Request Handling

When Alertmanager sends a POST to `/alert`:

1. Read and parse the JSON body (Alertmanager's grouped alert payload)
2. Append the full payload as a single JSON line to `/var/log/alerts.log`
3. For each alert in the `alerts` array:
   - Extract `alertname`, `severity`, `status`, and `instance` from labels
   - Write an evidence file named: `alert_{alertname}_{status}_{timestamp}.json`
   - Print a structured log line to stdout
4. Return HTTP 200 with `{"status": "ok", "received": "<timestamp>"}`

### GET Endpoint

A GET request to `/alert` (or any path) returns operational status:

```json
{
  "status": "running",
  "total_alerts_received": 5,
  "log_file": "/var/log/alerts.log",
  "evidence_dir": "/app/evidence"
}
```

### Evidence File Format

Each evidence file contains:

```json
{
  "received_at": "2026-04-05T02:44:11.185348Z",
  "alert_name": "ServiceDown",
  "severity": "critical",
  "status": "firing",
  "full_alert": {
    "status": "firing",
    "labels": {
      "alertname": "ServiceDown",
      "instance": "app2:5000",
      "job": "flask-app",
      "severity": "critical"
    },
    "annotations": {
      "description": "...",
      "summary": "..."
    },
    "startsAt": "2026-04-05T02:44:06.16Z",
    "endsAt": "0001-01-01T00:00:00Z",
    "generatorURL": "...",
    "fingerprint": "72330006fdb4c6ee"
  }
}
```

### Deployment

The webhook receiver runs on port 9094 inside the Docker network. Alertmanager sends alerts to `http://webhook-receiver:9094/alert`. The evidence directory is mounted as a volume to persist files across container restarts.

## 7. Alert Response Times

### Theoretical Response Time Model

The time from a failure occurring to the webhook receiver receiving the alert depends on several factors:

```
Total Response Time = Scrape Interval + For Duration + Group Wait + Network Latency
```

#### For ServiceDown (critical):

| Component | Duration | Notes |
|---|---|---|
| Scrape interval | 0-15s | Worst case is the full scrape interval if the failure happens right after a scrape |
| For duration | 15s | Alert must remain true for 15 consecutive seconds |
| Group wait (critical) | 5s | Alertmanager batching window |
| Network (internal Docker) | <100ms | Container-to-container within same Docker network |
| **Theoretical worst case** | **~35s** | |
| **Theoretical best case** | **~20s** | Failure happens just before a scrape |

#### For HostHighCpuUsage (warning):

| Component | Duration | Notes |
|---|---|---|
| Scrape interval | 0-15s | Node exporter uses default 15s interval |
| For duration | 5m (300s) | CPU must be sustained high for 5 minutes |
| Group wait (warning) | 15s | Longer batching window for warnings |
| **Theoretical worst case** | **~330s** | |

### Observed Response Times

| Alert | Failure Time | Alert `startsAt` | Webhook Received | Failure-to-Webhook |
|---|---|---|---|---|
| ServiceDown (app2) | 02:42:52Z | 02:44:06Z | 02:44:11Z | **79 seconds** |
| ServiceDown (app5) | ~02:48:18Z* | 02:49:21Z | 02:49:26Z | **~68 seconds** |
| HostHighCpuUsage | N/A (gradual) | 02:50:49Z | 02:51:04Z | **~15 seconds** (from startsAt to webhook) |

*app5 was killed as part of the multi-instance kill at 02:48:18Z, but Prometheus may not have been scraping app5 in the deployment configuration at that moment (the prometheus.yml targets only app1-app3).

The observed 79-second response time for the app2 ServiceDown alert includes:
- ~74 seconds from kill (02:42:52Z) to alert firing (02:44:06Z): this exceeds the theoretical maximum, which suggests the Prometheus evaluation cycle timing aligned unfavorably, or the `for` counter was reset by a brief successful scrape.
- ~5 seconds from alert firing to webhook receipt (02:44:06Z to 02:44:11Z): this matches the 5-second `group_wait` for critical alerts exactly.

## 8. Alert Resolution Tracking

### How Resolved Alerts Work

1. **Prometheus evaluation:** On each evaluation cycle (every 15 seconds), Prometheus evaluates all alert rules. When a previously-firing rule's expression becomes false (e.g., `up{instance="app2:5000"}` returns to 1), Prometheus marks the alert as resolved and sets the `endsAt` timestamp.

2. **Alertmanager notification:** Because `send_resolved: true` is configured on the webhook receiver, Alertmanager sends a notification with `"status": "resolved"` when it receives the resolution from Prometheus.

3. **Webhook receiver logging:** The webhook receiver processes resolved alerts identically to firing alerts -- writing an evidence file with `_resolved_` in the filename and appending to the log.

4. **Fingerprint correlation:** Every alert has a `fingerprint` field that remains constant across the firing and resolved lifecycle. This is the key for correlating a resolved alert back to its original firing alert:

   ```
   Firing:   fingerprint=72330006fdb4c6ee, startsAt=02:44:06Z, endsAt=0001-01-01T00:00:00Z
   Resolved: fingerprint=72330006fdb4c6ee, startsAt=02:44:06Z, endsAt=02:47:06Z
   ```

   The `endsAt` value of `0001-01-01T00:00:00Z` in the firing alert is a sentinel meaning "still firing" (Go's zero time). Once resolved, this is updated to the actual end time.

### Resolution Evidence Captured

| Firing Alert | Resolved Alert | Duration | Fingerprint |
|---|---|---|---|
| ServiceDown app2 @ 02:44:06Z | Resolved @ 02:47:06Z | 3m 0s | 72330006fdb4c6ee |

### Auto-Resolution Timeout

If Prometheus stops sending a firing alert (e.g., Prometheus itself crashes), Alertmanager will auto-resolve the alert after the `resolve_timeout` of 5 minutes. This prevents stale alerts from remaining in firing state indefinitely when the monitoring system itself has issues.
