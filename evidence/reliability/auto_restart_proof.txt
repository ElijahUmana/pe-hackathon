=== CHAOS EXPERIMENT: CONTAINER FAILURE + FAULT TOLERANCE ===
Date: 2026-04-05

Goal: Prove the system survives container failure without downtime.
Method: Kill one app instance (app2) and verify service continuity.

=== BEFORE (All 3 app instances healthy) ===

=== KILL app2 at Sun Apr  5 07:09:43 AM UTC 2026 ===

=== AFTER 120s (NO MANUAL INTERVENTION) ===
app2 remains exited (exit code 137 = SIGKILL):
NAMES                 STATUS
pe-hackathon-app2-1   Exited (137) 2 minutes ago

Service STILL HEALTHY — zero downtime despite 1/3 instances being dead:

{

  "status": "ok"
}

=== DOCKER COMPOSE RECOVERY ===
$ docker compose up -d app2
 Container pe-hackathon-app2-1 Starting
 Container pe-hackathon-app2-1 Started

All 3 instances back to healthy within 10 seconds:
NAME                              STATUS
pe-hackathon-app1-1               Up 51 minutes (healthy)
pe-hackathon-app2-1               Up 10 seconds (healthy)
pe-hackathon-app3-1               Up 53 minutes (healthy)

=== CONCLUSION ===
1. Multi-instance architecture (3 app containers behind nginx) provides fault tolerance
2. When one instance fails, nginx automatically routes traffic to surviving instances
3. Health checks continue to pass — zero user-facing downtime
4. Recovery via docker compose is instant
5. All containers configured with restart: unless-stopped policy
